# クエリ拡張・同義語処理（日本語の表記ゆれ対応）

## 課題

現在の FTS（bigram full-text search）はキーワードの完全一致依存。
ベクター検索がセマンティックな揺れをある程度吸収しているが、以下は拾えない。

| パターン | 例 |
|---|---|
| 英字表記ゆれ | ウェブ / Web / WEB |
| 略語・正式名称 | AI / 人工知能 / エーアイ |
| 活用形 | 走る / 走った / 走り |
| 送り仮名 | 書き込み / 書込み |

## 実装の方向性（案）

**案 A: 形態素解析 + 辞書拡張**
- `fugashi`（MeCab Python バインディング）または `sudachipy` を導入
- クエリを形態素に分解 → 基本形に正規化 → FTS に渡す
- 同義語辞書（ユーザー定義 YAML）でクエリ拡張
- 難易度: 高（形態素解析器の依存追加、辞書整備コスト）

**案 B: LanceDB の FTS オプションを活用**
- LanceDB の FTS インデックス設定（`with_position=True` など）でトークナイズを調整
- 現時点では日本語向けの選択肢が限られる

**案 C: LLM によるクエリリライト（MCP 利用時限定）**
- MCP ツール呼び出し前に AI クライアント側でクエリを正規化させるプロンプトを `prompts.py` に追加
- CLI 単体では機能しないが、MCP ユースケースには有効

## 保留理由

- 形態素解析器の依存追加はパッケージサイズと起動速度に影響
- ベクター検索がどこまでカバーできるか、実使用データで検証してから判断したい
- まず JSON 出力でエージェント連携を強化し、ユースケースが増えてから優先度を再評価
